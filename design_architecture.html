1. System Architecture Overview

The Vedic Astrology Platform is built as a cloud-native, microservices-based system deployed via container orchestration (Kubernetes). Each service is independently deployable, stateless, and horizontally scalable. The architecture is cleanly separated into six logical layers: Client Layer → API Gateway → Core Services → Data Layer → Infrastructure Layer → External Integrations. Per project requirements, the payment gateway, dasha calculation engine, and UI are fully isolated modules.

🖥️

Cross-Platform Clients

React SPA (Web) + React Native (iOS/Android). Shared design system and i18n.

🔀

API Gateway

Single entry point for all clients. JWT auth, rate limiting, routing, WAF.

⚙️

7 Core Microservices

Auth, Kundli Engine, Dasha Engine, AI Interpretation, Credit Wallet, Payment, Analytics.

🗄️

Polyglot Data Layer

PostgreSQL (transactional), Redis (cache/sessions), S3 (reports), OpenSearch (analytics).

🔒

Security First

TLS 1.3, WAF, KMS, OWASP Top 10 hardened, PCI-DSS scope minimised via gateway-hosted card entry.

🔗

External Integrations

LLM provider (OpenAI / Anthropic), Swiss Ephemeris, Payment Gateway (Razorpay / Stripe), Firebase Push.

3. Microservice Definitions

Each service owns its data, exposes REST APIs, and communicates with other services via the API Gateway or internal message queue for async operations.

🔐

auth-service

Handles registration, login, OAuth (Google/Apple), 2FA/MFA, JWT issuance, session management.

  • Access token: 15 min TTL
  • Refresh token: 7 day TTL (rotated)
  • Bcrypt password hashing
  • TOTP-based 2FA
Node.jsPostgreSQLRedis
🔭

kundli-engine

Core astrological calculation engine. Computes Nirayan chart, divisional charts, Shadbala, yogas, Ashtakavarga.

  • Swiss Ephemeris integration
  • Lahiri / Raman / KP Ayanamsa
  • North + South Indian chart styles
  • D1–D16 divisional charts
PythonFastAPIpyswisseph
📅

dasha-engine

Computes all Dasha systems: Vimshottari, Yogini, Chara, Kalachakra, Narayana, Moola. Pluggable per-dasha modules.

  • Mahadasha → Sookshma levels
  • Timeline JSON output
  • Transit (Gochara) correlation
  • Pluggable module interface
PythonFastAPIPostgreSQL
🤖

ai-interpretation-service

Receives structured JSON from kundli/dasha engines, constructs locale-aware prompts, calls LLM, validates output.

  • Anonymised prompts (no PII)
  • Deterministic validation layer
  • Conversational follow-up ("Ask Astrologer")
  • 3 report types: Personal / Compatibility / Career
PythonFastAPIRedis cache
💳

credit-wallet-service

Manages credit wallet lifecycle: balance checks, atomic deductions, refunds, promotional credits, ledger history.

  • Atomic credit transactions (DB tx)
  • Promo credits consumed first
  • Auto-refund on AI failure (≤60s)
  • Low-balance alert trigger (≤3 credits)
Node.jsPostgreSQLSQS
💰

payment-gateway-service

PCI-DSS-scoped payment module. Handles card (3DS), UPI (VPA/QR), webhook receipt, geo-based pricing.

  • Razorpay / Stripe adapters
  • Gateway-hosted card iframe (no card data on servers)
  • UPI deep-link intent flow
  • Idempotent webhook handler
Node.jsPostgreSQLWebhooks
📄

report-export-service

Generates PDF, DOCX, JSON/CSV exports, shareable expirable links, and print-optimised layouts.

  • PDF via WeasyPrint / Puppeteer
  • DOCX via python-docx (Pro users)
  • Signed S3 presigned URLs
  • Link expiry configurable
PythonS3WeasyPrint
📊

analytics-service

Powers the Super Admin dashboard. Aggregates anonymised user traffic, feature usage, revenue, and system health metrics.

  • OpenSearch for metrics storage
  • Real-time + historical views
  • CSV/Excel export
  • Weekly automated email report
PythonOpenSearchCelery
🔔

notification-service

Dispatches push notifications (FCM), emails (SES), and in-app alerts for Dasha transitions, low balance, and receipts.

  • Async via message queue
  • i18n-aware templates
  • Push + email + in-app
Node.jsFirebase FCMAWS SES

9. Security Architecture

Security is non-negotiable per SRS §5.7. The following controls map directly to the SRS critical security requirements.

🗄️

Database Security (§5.7.1)

  • 🔒Private subnet; no public DB ports exposed
  • 🔑AES-256 at rest; KMS-managed keys
  • 👤Least-privilege DB roles per service
  • 🛡️ORM/parameterised queries only; no raw SQL concat
  • 📋Immutable audit log; 12-month retention
  • 🔄Daily encrypted backups; quarterly restore test
  • 🚫No production data in dev/test environments
🔧

Application Security (§5.7.2)

  • 📋OWASP Top 10 audit before each phase go-live
  • 🔐JWT on all endpoints; unauthenticated = register/login/public dir only
  • ⏱️Rate limiting on all endpoints; lockout on repeated auth failures
  • ✅Server-side input validation on ALL user data
  • 🔑Secrets in Secrets Manager vault; never in code or env plain text
  • 🔍CVE dependency scanning in CI/CD pipeline
🌐

Network Security (§5.7.3)

  • 🔐TLS 1.3 minimum; TLS 1.0/1.1 disabled; HTTP rejected
  • 🛡️WAF deployed in front of all public services
  • ☁️DDoS protection at edge (CloudFront/Cloudflare)
  • 🏗️Network segmentation: public edge → app tier → data tier
  • 📝IaC review for all infra changes; no ad-hoc production changes
🔑

Identity & Access (§5.7.5)

  • 📱MFA available to all users; mandatory for admins and astrologers
  • ⏱️Access token: 15 min; Refresh token: 7 days; invalidated on logout/password change
  • 👥Quarterly admin access review
  • 📋All privileged admin actions require 2FA + audit log entry
  • 💳PCI-DSS scope: card data never touches platform servers (gateway-hosted iframe)
🧪

Security Testing & Incident Response (§5.7.6)

Third-party pentest before Phase 1 go-live (critical/high findings must be remediated first). Automated vulnerability scan on every production deployment. Documented incident response plan (detect → contain → eradicate → recover → notify). Breach notification to affected users within 72 hours (GDPR Phase 2+). AI LLM prompts must be anonymised — no PII in raw LLM API calls.

11. Deployment & Infrastructure

All infrastructure is defined as code (Terraform). Deployments use Helm charts on Kubernetes. No manual changes to production are permitted.

☁️

Cloud & Kubernetes

AWS (primary) or GCP. EKS for container orchestration. Each microservice in its own Deployment + Service. HPA for autoscaling. External Secrets Operator syncs secrets from Secrets Manager.

🔀

CI/CD Pipeline

GitHub Actions: build → lint → unit tests → dependency CVE scan → Docker build → push to ECR → Helm deploy. Rolling updates with readiness probes. Automated rollback on failure.

📊

Observability

Prometheus metrics scraped from each service. Grafana dashboards for p50/p95/p99 latency, error rates, pod health. CloudWatch log aggregation. PagerDuty alerting on SLA breach.

🌍

Network Zones

Zone A (Public): CloudFront CDN + WAF + ALB. Zone B (App): K8s pods — API Gateway + microservices. Zone C (Data): RDS PostgreSQL + ElastiCache Redis (private subnet, no public access).

🔄

Environments

Three environments: Dev (feature branches), Staging (integration + pentest), Production (blue/green deploy). No prod data in dev/staging. Synthetic data sets for testing.

📋

IaC & Compliance

Terraform for all cloud resources. Terraform plan reviewed in PR before apply. GDPR-compliant data residency tagging. DPDP Act compliance for Indian user data. PCI-DSS scope limited to payment-gateway-service only.

12. Scenario 1 — Email Verification, Wallet Economics & Chat Billing

Covers the full lifecycle from registration to first paid interaction: email ownership proof, dollar-cent wallet model, per-report and per-chat-message billing.

Email Service Module

Location: backend/auth/email_service.py

SMTP config (via .env):

  • EMAIL_HOST — e.g. smtp.gmail.com
  • EMAIL_PORT — 587 (STARTTLS)
  • EMAIL_USER — sender address
  • EMAIL_PASSWORD — app password
  • EMAIL_FROM_NAME — display name
  • APP_BASE_URL — for link generation

Mock mode (no SMTP): prints link to stdout.

🔐

Token Security

  • Token: uuid.uuid4() — 128-bit random
  • TTL: 24 hours from registration
  • One-time use: cleared immediately on verification
  • Resend endpoint: POST /auth/resend-verification
  • Rate-limited: max 3 resend requests per hour per email
  • Unverified users can still log in but see banner prompting verification

12.3 Chat Billing — AI Interpreter

💬

Per-Message Deduction

Before each AI chat response, 5 cents are deducted from the user's wallet. If balance < 5 cents, API returns HTTP 402 with message: "Insufficient balance. Top up to continue chatting."

🔄

Refund on AI Error

If the AI call fails after deduction (network error, timeout, content filter), the 5-cent charge is automatically refunded. A wallet_transaction log entry records both the debit and the refund.

👤

Guest Users

Guest sessions bypass all billing checks. Guests have no wallet and cannot top up. After 3 free AI responses (configurable via GUEST_FREE_CHAT_LIMIT), guests see an upsell prompt to register.

13. Scenario 2 — Super Admin Panel

A server-rendered HTML panel accessible only to platform administrators. Credentials are set via environment variables — no database table required.

🔑

Authentication

  • HTTP Basic Auth via FastAPI HTTPBasic dependency
  • Credentials: ADMIN_USERNAME and ADMIN_PASSWORD in .env
  • Constant-time comparison prevents timing attacks
  • All admin routes return 401 WWW-Authenticate: Basic on failure
  • No session tokens — each request re-authenticates
  • Admin routes are under /admin/* prefix, excluded from public API schema
📋

Panel Features

  • User list: email, name, verified status, wallet balance, created date
  • Read-only SQL query runner (SELECT only; non-SELECT blocked)
  • Application log viewer (last 500 lines of server log)
  • Manual wallet credit/debit per user
  • Force email verification for a user
  • Delete user account
  • System stats: total users, total credits issued, readings served today

🛡️

Security Constraints

  • SQL runner: SELECT-only; blocked keywords: INSERT, UPDATE, DELETE, DROP, ATTACH
  • Log viewer: read-only; no shell command execution
  • All admin actions are written to a separate admin_audit.log
  • Admin panel not linked from public UI
  • Brute-force protection: lockout after 5 failed attempts per IP (15-min window)
📊

Dashboard Stats

  • Total registered users
  • Verified vs unverified count
  • Readings generated today / this week
  • Total wallet credits issued (cents)
  • Active sessions (last 24 hours)
  • Top-up revenue (cents collected)
⚙️

Env-var Config

  • ADMIN_USERNAME — admin login name
  • ADMIN_PASSWORD — strong password (min 16 chars)
  • ADMIN_LOG_PATH — path to server log file
  • ADMIN_AUDIT_LOG — path for admin action audit log
  • No default — panel is disabled if vars not set
