1. Executive Summary

This audit compares the current NarayanAstroReader codebase against the Phase-1 requirements defined in the design_architecture.html specification. The system is working and functional as a monolithic prototype but deviates significantly from the target microservices architecture in terms of folder structure, service isolation, database technology, and several functional capabilities.

5. Remediation Plan — Prioritised

📌 Approach Items are ordered by production-readiness impact. Critical items block go-live; High items are needed before Phase-2; Medium/Low can be deferred.
#GapPriorityAction Required
1 Microservice folder structure CRITICAL Create services/ directory; move each module into its own subfolder with README.md, Dockerfile, and requirements.txt. (Being addressed in this session.)
2 README / documentation in each service CRITICAL Each service folder needs: README.md (purpose, API endpoints, env vars, how to run), inline docstrings, API contract docs.
3 JWT-based authentication CRITICAL Replace email+session_id with JWT Bearer tokens. Implement access token (15 min) + refresh token (7 days) flow. Required by architecture §9 and all service-to-service auth.
4 PostgreSQL migration HIGH Replace SQLite with PostgreSQL. Migrate schema, add connection pool (asyncpg / psycopg3). Required for production horizontal scaling.
5 Wallet ledger — separate DB table HIGH Create immutable wallet_ledger table (id, wallet_id, type, credits_delta, balance_after, created_at). Replace JSON-in-column approach.
6 Paid balance vs Promo balance HIGH Add paid_balance + promo_balance columns. Promo credits consumed first. Required by architecture DB schema §6.
7 Dasha Engine — standalone service HIGH Create services/dasha-engine/ with /dasha/calculate and /dasha/current/{profile_id} endpoints. At minimum expose Vimshottari as structured API. Add stub modules for Yogini/Chara per pluggable interface.
8 Auto-refund on AI failure (≤60s) HIGH In the streaming endpoint: on exception or timeout, call wallet credit-back. Architecture NFR mandates this for user trust.
9 Razorpay / Stripe real SDK integration HIGH Activate razorpay Python/Node SDK. Implement real order creation, HMAC webhook verification, and idempotent credit grant after payment.
10 Dockerfile for each service HIGH Each service needs a multi-stage Dockerfile. Add docker-compose.yml for local dev spanning all services.
11 Compatibility + Career report types MEDIUM Add report_type: "compatibility" | "career" to AI interpretation service. Write prompt templates for each.
12 Password reset / forgot-password flow MEDIUM Add /auth/forgot-password (sends reset email) and /auth/reset-password (consumes token) endpoints.
13 OAuth — Google + Apple login MEDIUM Implement OAuth2 callback endpoints. For web: redirect flow. For mobile: Expo AuthSession.
14 Low-balance alert trigger MEDIUM After each deduction, check if total balance ≤ threshold (e.g. 3000 paise / ₹30). If so, send notification (email or in-app) via notification-service.
15 DOCX report export MEDIUM Add python-docx based report generator in services/report-export-service/. Expose as POST /export/{report_id}/docx.
16 S3 report storage + shareable links MEDIUM Upload generated PDFs to S3. Return presigned URL. Add /export/{report_id}/share endpoint with configurable expiry (24h default).
17 Redis cache layer MEDIUM Add Redis for session tokens, rate-limiting counters, and kundli calculation caching. Required per architecture tech stack §8.
18 SVG/graphical kundli chart MEDIUM Render proper North/South Indian chart SVG in frontend. Currently text-only representation.
19 2FA / MFA (TOTP) MEDIUM Add speakeasy (or pyotp) TOTP-based 2FA. Mandatory for admin accounts per architecture §9.
20 Analytics service LOW Create services/analytics-service/ with Celery tasks. Metrics: DAU/MAU, revenue, feature usage, API latency. Phase 2 can use simpler aggregation first.
21 Push notifications (Firebase FCM) LOW Implement notification-service with FCM for mobile push. Required for Phase 2 mobile app.
22 Async message queue (SQS) LOW Replace synchronous email sending with SQS consumer. Required for horizontal scaling.
