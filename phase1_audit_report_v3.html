15. Prioritised Gap List

🔴 Critical — Must fix before Phase 1 launch

  • CRITICAL No MFA / 2FA anywhere in the system
    SRS §5.7.5 makes MFA mandatory for all admin accounts. SRS §3.1.1 requires 2FA for general users. No TOTP/speakeasy implementation. Admin portal uses single-factor secret key only.
  • CRITICAL PII not encrypted at rest
    SRS §5.2 + §5.7.1 require AES-256 encryption at rest for name, DOB, TOB, birth place, email. All stored as plaintext in PostgreSQL. No KMS integration. Hard-coded JWT_SECRET and ADMIN_SECRET defaults violate §5.7.2.
  • CRITICAL No JWT refresh token / logout token invalidation
    SRS §5.7.5 requires refresh token (7-day) with immediate invalidation on logout/password change. Only access tokens issued. No /auth/refresh, no /auth/logout endpoint, no token blacklist.
  • CRITICAL No OAuth (Google / Apple) login
    SRS §3.1.1 requires social OAuth as a registration option. No Google/Apple buttons, no Passport.js/OAuth flow, no oauth_provider column in users table.
  • CRITICAL No CI/CD pipeline — no CVE scanning, no automated tests on deploy
    SRS §5.7.2 requires CVE dependency scanning in CI/CD. No GitHub Actions, no automated test runner, no Docker build pipeline. Vulnerable dependencies could be shipped undetected.

🟡 High — Required for a production-ready Phase 1

  • HIGH No SVG/Canvas kundli chart rendering (North/South Indian styles)
    SRS §3.2.1 requires visual chart display in both North Indian diamond and South Indian square styles. Currently only text/JSON output. Core user-facing feature gap.
  • HIGH Analytics dashboard is basic — no real metrics
    SRS §3.10 requires DAU/MAU, feature usage, revenue analytics, system health (p50/p95/p99), date-range filters, CSV export, weekly digest. Current admin shows only 4 stat cards. analytics-service scaffold is empty.
  • HIGH Missing report types: Yearly Forecast, Remedies
    SRS §3.4.1 requires Yearly Forecast (Gochara + Dasha overlay) and Remedies report (gemstone, mantra, yantra). Neither endpoint nor AI prompt exists.

16. Summary & Recommendations

What is working well ✅

The core AI reading pipeline (kundli → deep reading → chat → career/compat reports) is functional end-to-end. The wallet system is architecturally sound with atomic transactions, promo-first logic, auto-refund on failure, and an immutable ledger. Multi-language (EN/HI/TA) works correctly with reactive switching. Rate limiting, account lockout, and JWT enforcement on all protected endpoints are now in place. The admin panel has secret-key authentication with brute-force lockout. Redis session store provides a foundation for horizontal scaling. PIN-protected shareable reading links with 72h expiry are fully working. AI disclaimer is present on all reports in both prompts and UI.

The top 3 decisions required before Phase 1 launch

1. Architecture path choice: Decide whether to (a) deploy the current monolith behind a reverse proxy and refactor services incrementally post-launch, or (b) wire up the services/ scaffolding before launch. Option (a) is realistic for Phase 1 given the timeline; document it as a known deviation.

2. PII encryption and secrets management: The hard-coded JWT_SECRET and ADMIN_SECRET defaults, and plaintext PII in the database, are the most critical security risks before handling real user data. Minimum: move all secrets to environment variables with validation on startup; add column-level encryption for name/birth data.

3. MFA and OAuth: SRS §3.1.1 and §5.7.5 make these non-optional. Integrate at minimum TOTP-based 2FA for admin accounts and Google OAuth as a registration option before going live.
