PARTIAL

Service Layer Compliance

Architecture specifies 9 distinct service layers. Status below reflects whether each has a running implementation (not just a scaffold).

✅ Presentation Layer — frontend/index.html is a functional SPA with all major screens, multi-language, and responsive design.
❌ API Gateway — No dedicated API Gateway service. backend/api/main.py acts as a monolith handling auth, routing, billing, and business logic — violating the architecture's single-responsibility design. All 37+ endpoints are in one 1342-line file.
✅ Calculation Service (Kundli Engine) — backend/kundli_engine/engine.py and services/kundli-engine/ are both implemented. PyJHora integration provides real ephemeris computation.
✅ AI Interpretation Service — backend/ai_interpretation/agent.py fully implemented with streaming, safety filter, planet calibrator, and 6 report types. services/ai-interpretation-service/ mirrors this.
⚠️ User & Profile Service — Embedded in backend/persistence/database.py and main.py. Functional but not extracted as a standalone service. No multi-profile support per user.
⚠️ Payment Service — payment/gateway.py has Razorpay + Stripe adapters but all return mock data. No real integration. No webhook handler. services/payment-gateway-service/ is a stub.
✅ Credit Wallet Service — Fully implemented: ledger table, paid/promo split, atomic debit (promo-first), auto-refund, geo-pricing. Both backend/persistence/database.py and services/credit-wallet-service/ are in sync.
❌ Analytics Service — services/analytics-service/ has Dockerfile + README only. src/ folder contains no Python files. No event tracking, metric aggregation, or dashboard data pipeline exists anywhere in the codebase.
❌ i18n Service — Architecture specifies a dedicated service serving language bundles. Translations are hardcoded in a JS object inside index.html. No i18n-service/ exists. Adding a new language requires code changes.
✅ Data Store — PostgreSQL with psycopg2 connection pool. 3 schema migration files. Docker Compose provisions Postgres 16. Redis provisioned but not yet used by app (sessions still in-memory dict).
⚠️ Dasha Engine — services/dasha-engine/ is fully implemented as a standalone FastAPI microservice with pluggable architecture. Only Vimshottari is implemented; 5 other SRS-required systems are stubbed.
⚠️ Report Export Service — services/report-export-service/ has PDF generator only. DOCX generator lives in backend/reports/ not replicated here. No S3/object storage integration for persisting reports.
CRITICAL GAP

Monolith vs. Microservices Architecture Mismatch

The codebase has two parallel implementations that are not synchronised:

Running Monolith: backend/api/main.py (1342 lines) contains all business logic, routing, auth, billing, AI calls, dasha, export, sharing, and admin. This is what actually runs via start.py.
Service Scaffolds: services/ directory has 9 sub-services with Dockerfiles, READMEs and some implementations. These are not wired together or deployed. The docker-compose.yml does not include the monolith backend.

Recommendation: For Phase 1 launch readiness, either (a) complete the monolith with all missing features and deploy it with Nginx reverse proxy, or (b) choose 2–3 services to properly extract (dasha-engine, report-export) and keep the rest in the monolith. Do not maintain both in parallel without a clear migration plan.
